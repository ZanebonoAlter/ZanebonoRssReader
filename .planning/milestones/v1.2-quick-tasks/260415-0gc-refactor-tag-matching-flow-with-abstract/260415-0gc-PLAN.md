# Quick Task 260415-0gc: Refactor Tag Matching Flow with Abstract Tag Hierarchy

**Created:** 2026-04-14
**Mode:** quick (no validation)

## Summary

Refactor the tag matching flow in `findOrCreateTag` to properly handle abstract tags with three-level thresholds, and add post-creation abstract tag similarity matching for multi-level hierarchy.

## Context

Current implementation does not distinguish between normal and abstract tags during matching. The new flow needs to:
1. Distinguish abstract tags (source="abstract") from normal tags during embedding matching
2. Handle parent-child relationships correctly based on threshold and tag type
3. Delete child tag embeddings after establishing parent-child relationships (prevent tag proliferation)
4. After creating abstract tags, immediately search for similar abstract tags for multi-level hierarchy
5. Use AI judgment (with new prompt) for abstract-vs-abstract middle-band matching

## Task 1: Refactor `findOrCreateTag` in `tagger.go`

**Files:**
- `backend-go/internal/domain/topicextraction/tagger.go`

**Action:**
Modify the `findOrCreateTag` function's `TagMatch` result handling to:

1. **Exact match** (no change): return existing tag directly.

2. **High similarity (>= 0.97)**:
   - If matched tag is **normal** (source != "abstract"): reuse as-is (no change).
   - If matched tag is **abstract** (source == "abstract"): create the new tag, establish parent-child relation (abstract=parent, new=child), delete new tag's embedding.

3. **Middle band (0.78 ~ 0.97)**:
   - If matched tag is **normal**: call `ExtractAbstractTag` to create abstract tag from the pair. After abstract tag is created, delete **both** child tags' embeddings (the matched normal tag's and the new tag's). Return the first child tag.
   - If matched tag is **abstract**: create new tag, establish parent-child relation (matched abstract=parent, new=child), delete new tag's embedding. No LLM call needed.

4. **Low similarity (< 0.78)**: create new tag + generate embedding (no change).

Key implementation detail: need to check `matchResult.ExistingTag.Source == "abstract"` to distinguish tag types. For middle-band normal, the `ExtractAbstractTag` function already handles LLM abstract name generation and parent-child relations. Need to add embedding deletion for both children after abstract creation.

**Verify:** `go build ./...` passes

**Done:** `findOrCreateTag` correctly routes to different handlers based on threshold + tag type combinations.

## Task 2: Add post-creation abstract tag similarity matching in `abstract_tag_service.go`

**Files:**
- `backend-go/internal/domain/topicanalysis/abstract_tag_service.go`

**Action:**
Add a new exported function `MatchAbstractTagHierarchy(ctx, abstractTagID)` that:

1. Loads the newly created abstract tag and its embedding.
2. Searches for similar **abstract-only** tags using pgvector, excluding self and tags already in a parent-child relationship.
3. Applies the same thresholds:
   - **>= 0.97**: new abstract tag becomes child of the existing abstract tag. Delete new abstract tag's embedding.
   - **0.78 ~ 0.97**: call LLM (new prompt) to determine parent-child direction. Establish the relationship. Delete the child's embedding.
   - **< 0.78**: no operation.
4. Uses existing `wouldCreateCycle` for safety.
5. Creates a new `buildAbstractHierarchyPrompt` for AI judgment between abstract tags. The prompt asks which concept is broader (parent) and which is more specific (child).

For the pgvector query, add a method to `EmbeddingService` or inline a query that filters `topic_tags.source = 'abstract'`.

Call this function at the end of `ExtractAbstractTag` after the transaction succeeds, passing the created abstract tag's ID.

**Verify:** `go build ./...` passes

**Done:** After creating an abstract tag, the system immediately checks if it should be linked into an existing abstract hierarchy.

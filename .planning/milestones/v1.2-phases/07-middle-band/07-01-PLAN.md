    ```go
        ChildID        uint      `gorm:"not null;uniqueIndex:idx_tag_relation_pair" json:"child_id"`
    ```

    ```sql
    CREATE TABLE IF NOT EXISTS topic_tag_relations (
      id SERIAL PRIMARY KEY,
      parent_id INTEGER NOT NULL REFERENCES topic_tags(id) ON DELETE CASCADE,
      child_id INTEGER NOT NULL REFERENCES topic_tags(id) ON DELETE CASCADE,
      relation_type VARCHAR(20) NOT NULL DEFAULT 'abstract',
      similarity_score FLOAT,
      created_at TIMESTAMP DEFAULT NOW(),
      UNIQUE(parent_id, child_id)
    );
    CREATE INDEX IF NOT EXISTS idx_tag_relations_parent ON topic_tag_relations(parent_id);
    CREATE INDEX IF NOT EXISTS idx_tag_relations_child ON topic_tag_relations(child_id);
    ```

    **3. 创建 abstract_tag_service.go**:
    - `ExtractAbstractTag(ctx, candidates []TagCandidate, newLabel string, category string) (*models.TopicTag, error)` — 核心函数
      - 构造 LLM prompt：给候选标签的 label + 相似度，要求返回共同抽象概念
      - 使用 `airouter.NewRouter().Chat()` 调用，Capability 使用 `CapabilityTopicTagging`（复用现有 topic_tagging route per D-01）
      - LLM 返回 JSON: `{"abstract_name": "...", "reason": "..."}`
      - 用 `topictypes.Slugify()` 生成 slug
      - 检查是否已有同 slug 的抽象标签（避免重复 per D-05 支持多级嵌套）
      - 如果已存在：复用，仅建立新的父子关系
      - 如果不存在：创建新 TopicTag (source="abstract") + embedding + 建立 parent-child relations
      - 为所有候选子标签建立 relation 记录 (relation_type="abstract", similarity_score=各自相似度)
      - 如果 LLM 调用失败：降级返回 nil（调用方会 fallback 到创建普通标签，per D-02 不阻塞入库）
      - category 继承自第一个子标签的 category（Claude's Discretion 决策）

    - `GetTagHierarchy(category string) ([]TagHierarchyNode, error)` — 层级查询
      - 查询所有 relation_type="abstract" 的记录
      - 构建树结构：parent 包含 children 列表，递归支持多级 (per D-05)
      - 每个 node 包含 tag info + children + 关联文章数

    - `UpdateAbstractTagName(tagID uint, newName string) error` — 更新名称 (per D-04)

      - 更新 label + slug（用 topictypes.Slugify）
      - 检查 slug 冲突
      - 重新生成 embedding（异步，enqueue）

    - `DetachChildTag(parentID, childID uint) error` — 分离子标签
      - 删除 topic_tag_relations 中对应记录
      - 如果 parent 不再有 children，不删除 parent（保留供历史追溯）

    在 `findOrCreateTag` 函数中，`case "ai_judgment":` 分支内：
    - 当前逻辑：直接 fall through 创建新标签
    - 修改为：调用 `topicanalysis.ExtractAbstractTag(ctx, matchResult.Candidates, tag.Label, category)`
    - 如果返回非 nil（抽象标签创建成功）：返回该抽象标签作为匹配结果，文章关联到抽象标签
    - 如果返回 nil（LLM 不可用或降级）：fall through 到原有的创建新标签逻辑
    - 日志记录：`[INFO] Abstract tag extracted: {name} from candidates [{labels}]` 或 `[WARN] Abstract tag extraction failed, falling back to new tag creation`

    **重要**：文章仍然关联到抽象标签（非子标签），因为 D-03 说"文章只关联到子标签"。但实际上新文章入库时，抽象标签是匹配结果，文章应关联到抽象标签本身用于聚合展示。D-03 的意思是抽象标签不替代子标签与文章的现有关联，新文章关联到抽象标签作为其标签。
    
    修正：按 D-03 "文章只关联到子标签，抽象标签仅用于展示和聚合"—— 新文章入库时，如果触发抽象标签提取：
    1. 创建抽象标签 + 建立子标签关系
    2. 文章关联到最近的子标签（即 matchResult 中最高相似度的候选标签），不关联到抽象标签
    3. 这样抽象标签纯粹作为聚合节点

<task type="auto">
  <name>Task 2: GetArticles 扩展关注标签筛选 + 相关度排序</name>
  <files>backend-go/internal/domain/articles/handler.go</files>
  <action>
扩展 GetArticles handler 支持关注标签筛选（per D-10 相关度=匹配标签数, D-11 全部关注混合流相关度排序, D-14 抽象标签展开子标签, D-15 抽象标签权重更高）:

1. 在 GetArticles 函数参数解析区域新增：
   ```go
   watchedTagIDs := c.Query("watched_tag_ids") // 逗号分隔的 tag ID 列表
   sortBy := c.Query("sort_by")                // "relevance" 或 "date"（默认 date）
   ```

2. 当 `watchedTagIDs` 非空时：
   - 解析为 `[]uint` 列表
   - 查询 `topic_tag_relations` 获取抽象标签（parent_id IN ids）的子标签 ID，合并到 tagIDs 列表中
   - JOIN `article_topic_tags` 并筛选 `article_topic_tags.topic_tag_id IN (expanded_tag_ids)`
   - 当 `sortBy == "relevance"` 时：
     - 计算每篇文章的匹配权重分数：抽象标签匹配权重 2，具体标签匹配权重 1
     - 需要区分哪些 tagIDs 是抽象标签的子标签（权重 2），哪些是直接关注的普通标签（权重 1）
     - 使用 `SELECT articles.*, SUM(CASE WHEN att.topic_tag_id IN (child_tag_ids) THEN 2 ELSE 1 END) AS relevance_score` 子查询
     - `ORDER BY relevance_score DESC, articles.pub_date DESC`
   - 当 `sortBy == "date"` 或为空时：`ORDER BY articles.pub_date DESC`（现有默认）

3. 在返回数据中增加 `relevance_score` 字段（当 sortBy=relevance 时），让前端知道相关度分数

4. 确保 COUNT 查询用 `COUNT(DISTINCT articles.id)` 避免因 JOIN article_topic_tags 导致数量膨胀

注意：watched_tag_ids 为空时，查询行为完全不变，不影响现有首页时间线。

**Core method — `scanAndMergeTags()`:**
This is the key logic. Use a single efficient SQL query to find similar tag pairs:

```sql
SELECT
    t1.id AS source_id, t1.label AS source_label,
    t2.id AS target_id, t2.label AS target_label,
    t1.category,
    e1.embedding <=> e2.embedding AS distance
FROM topic_tag_embeddings e1
JOIN topic_tags t1 ON t1.id = e1.topic_tag_id
JOIN topic_tag_embeddings e2 ON e2.topic_tag_id > e1.topic_tag_id
JOIN topic_tags t2 ON t2.id = e2.topic_tag_id
WHERE (t1.status = 'active' OR t1.status = '' OR t1.status IS NULL)
  AND (t2.status = 'active' OR t2.status = '' OR t2.status IS NULL)
  AND t1.category = t2.category
  AND e1.embedding <=> e2.embedding < ?
ORDER BY e1.embedding <=> e2.embedding ASC
LIMIT 50
```

The `e2.topic_tag_id > e1.topic_tag_id` condition prevents duplicate pairs and self-matches. The `?` placeholder is `1.0 - DefaultThresholds.HighSimilarity` (i.e., distance < 0.03 for 0.97 similarity). The LIMIT 50 caps work per cycle.

For each returned pair:
1. Load both tags from DB
2. Count articles for each: `SELECT COUNT(*) FROM article_topic_tags WHERE topic_tag_id = ?`
3. The tag with MORE articles becomes the **target** (keep it). The tag with FEWER articles becomes the **source** (gets merged). If equal, use the one with the smaller ID as source (deterministic).
4. Call `topicanalysis.MergeTags(sourceID, targetID)`
5. Log: `"[AUTO-MERGE] merged tag '%s' (id=%d, %d articles) into '%s' (id=%d, %d articles), similarity=%.4f"`
6. Track in RunSummary: increment MergedCount, append to MergeDetails

**Important implementation details:**
- The similarity threshold should use `topicanalysis.DefaultThresholds.HighSimilarity` (0.97), so distance threshold = 1 - 0.97 = 0.03
- After each merge, the merged tag's embedding is deleted and it's marked as "merged" — so subsequent pairs in the same run that reference the merged tag will fail. Wrap each merge in a check that both tags are still active before calling MergeTags. If either has status='merged', skip (increment SkippedCount).
- The `e2.topic_tag_id > e1.topic_tag_id` JOIN condition ensures we only get unique pairs (no A→B and B→A).

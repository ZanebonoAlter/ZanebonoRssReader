<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → queue APIs | Query params and retry requests are untrusted input entering queue handlers |
| merge transaction → async worker | Post-merge queueing crosses from committed relational state into eventual background processing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-260413-01 | T | merge re-embedding retry endpoint | mitigate | Validate limit/offset/status like existing queue handlers and only reset failed rows, never arbitrary task states |
| T-260413-02 | D | duplicate queue creation after repeated merges | mitigate | Enforce pending/processing dedupe in `Enqueue` for the surviving target tag before inserting a new task |
| T-260413-03 | I | worker reading inconsistent merge state | mitigate | Enqueue only after `MergeTags` transaction commits so the worker never processes half-migrated tag relationships |
| T-260413-04 | R | operator cannot audit why a re-embedding task exists | mitigate | Persist `source_tag_id` + `target_tag_id` and expose them in task list responses and frontend table |
</threat_model>
